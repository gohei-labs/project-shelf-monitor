ガードレール地図のグループ化

この成果物は、ガードレール地図を柱→カテゴリでグループ化してレジストリが 100〜200 件に育っても読めるようにすること、および category をレジストリで強制されるフィールドにすることの承認を求めます。

承認のお願い（Requested Approval）

  1. playbook/registry.json に categories マップと各ゲートの category フィールドを追加する（3 本柱にまたがる 10 カテゴリ）。
  2. ガードレール地図を「柱→カテゴリ」の折りたたみグループ、グループごとの件数、各行の実行場所バッジ、検索ボックス＋実行場所フィルタで再生成する。
  3. scripts/check-registry.ps1 がカテゴリ整合（存在・既知・二言語）を強制し、新フィールドが黙って乖離しないようにする。

リスク（Risk）

カテゴリ名と割当は編集上の判断だが、レジストリのデータであり、生成器に触れずに改名・再分類できる。検索とフィルタはプログレッシブ・エンハンスメントで、JavaScript 無効でも地図は全グループを開いた状態で読める。ゲートの挙動は変わらない — 強制されるガードレールは同じで、変わるのは見せ方（グループ化）だけ。

用語チェック（Terminology Check / 標準用語チェック）

用語意味
柱（Pillar）最上位の 3 グループのいずれか：ガードレール／可視化／人間オーケストレーション。
カテゴリ（Category）柱の中のより細かい機能的グループ（例：review-artifacts、bilingual-docs）。
実行場所（Venue）ガードレールがどこで走り、ブロックするか：ローカル＋CI／CI／条件付き／名目／方針／外部。
プログレッシブ・エンハンスメントページは JavaScript 無しでも機能し、スクリプトは検索とフィルタを足すだけ。

ランタイム境界チェック（Runtime Boundary Check）

本変更はドキュメント・レジストリ・生成器の変更である。ガードレールの「見せ方」を変えるだけで、どこで・どう走るかは変えない。実行場所ラベルは既存のランタイム（PowerShell によるローカル事前チェック、CI、Dev Container）を説明するもので、新たなランタイムは導入しない。

触れたツールは PowerShell の地図生成器のみ。Node は検証中に埋め込みフィルタスクリプトを構文チェック（node --check）するためだけに使い、新しいランタイム依存ではない。権威ある環境は引き続き Dev Container と Docker Compose。
