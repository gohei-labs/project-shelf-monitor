Monitor Design Refresh

RecommendedMonitorGenerated outputDesign

Decision Summary

Refresh the generated monitor pages so they read as compact operational dashboards rather than editorial landing pages.

What The Owner Needs To Approve

Approve the monitor design direction: the guardrail and review pages prioritize scan density, status filtering, generated-source traceability, and reliable readability over presentation-style animation.

Recommendation

Proceed. The new surface better matches the monitor concept: a place to inspect the current rule state quickly, not a marketing or narrative page.

Risks And Mitigations

RiskMitigation
The monitor becomes too dense for first-time readers.Keep the compact summary blocks at the top and preserve modal detail for each guardrail.
Status filtering hides important rules accidentally.The default filter is all rules, and the visible count updates with each filter.
Generated HTML diverges from the generator.The monitor and reviews pages are regenerated from the PowerShell generators and verified by the existing up-to-date checks.
Mobile text clips in narrow viewports.The Japanese lead copy uses explicit short lines and mobile wrapping rules.

What Is Not Changed

This does not change guardrail definitions, enforcement behavior, repository visibility, CI policy, or the public mirror publishing boundary.

The runtime boundary stays unchanged. The monitor is still generated by existing PowerShell scripts, checked by existing CI, and published through the existing GitHub Pages/public mirror workflow. No package manager, server runtime, Docker dependency, or local environment requirement is added.
