Editorial Guardrail Map

This review asks for approval to redesign the generated guardrail map as a crafted, near-monochrome editorial page - a masthead plus a typeset registry - while keeping it fully generated from the registry and still gated against drift.

Requested Approval

  1. Rewrite scripts/build-guardrail-map.ps1 to generate an editorial page from playbook/registry.json: a masthead (hero, three pillars, oversized numerals for 25 / 10 / 16) above a typeset registry of all rules.
  2. Adopt a near-monochrome palette with one rationed accent, Space Grotesk + Inter from Google Fonts, light/dark via prefers-color-scheme, and a single load-time motion that honors reduced-motion.
  3. Replace the collapsible groups and venue-chip filter with always-open sections plus one text search. The data and integrity are unchanged - check-registry.ps1 still blocks any registry/map drift.

Risk

The page now references Google Fonts at view time; if fonts.googleapis.com is blocked, it falls back to system fonts (still readable) - no build or runtime dependency is added. Venue/status chip filtering was dropped in favor of one text search; it can return in the editorial style if wanted. The change is presentation only and does not alter what is enforced.

Runtime Boundary Check

This change is to the map generator (PowerShell) and its generated output only. It introduces no build or runtime dependency: the maps are still static HTML copied into the docs portal and served as-is, and the local pre-check, CI check job, and Dev Container Bash lane all run the same check.ps1.

The one new view-time reference is the Google Fonts stylesheet (fonts.googleapis.com), loaded by the browser when the page is opened. It is optional - the page degrades gracefully to system fonts if it is blocked. The authoritative environment remains the Dev Container plus Docker Compose.
