PR Live Monitor Preview

RecommendedMonitorMobile reviewPublic preview

Decision Summary

Publish each pull request monitor preview to the public mirror under pr/<number>/ and add a stable PR comment with direct links.

What The Owner Needs To Approve

Approve public, unmerged monitor previews for pull requests. The preview is generated output only and does not overwrite the production monitor root.

Recommendation

Proceed. A normal URL is much easier to review on mobile than downloading an Actions artifact, and it keeps approval-before-merge visible.

Risks And Mitigations

RiskMitigation
Unmerged generated output becomes public.Publish only generated monitor site files, under pr/<number>/, using the existing public mirror boundary.
Publish credentials are exposed to PR-modified scripts.The PR workflow only uploads an artifact. A trusted workflow_run workflow from the default branch downloads that artifact and publishes it without executing PR code.
A PR preview overwrites the production monitor.The workflow passes a scoped target path; the publisher preserves the mirror root for PR previews.
The PR receives duplicate preview comments.The workflow updates an existing marker comment when one exists.
Preview publish cannot run because the deploy key is unavailable.The Actions artifact remains as a fallback preview.
Rejected PR previews remain public.Record this as a remaining risk; add cleanup on PR close as a follow-up if needed.

What Is Not Changed

This does not make the private source repository public, change guardrail source of truth, or deploy PR previews to the production root URL.

Terminology Check

TermUse here

Runtime Boundary Check

The runtime remains GitHub Actions, PowerShell, git, and the existing public mirror repository. The preview workflow adds PR comment writes through the GitHub API and does not add a package manager, server runtime, Docker dependency, or new hosting provider.
