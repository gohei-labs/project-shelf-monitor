Application Monitor Baseline

application code を入れる前に、product 自体を監視する別 monitor を追加する変更です。

Decision Summary

`product/application-map.json` を requirements、planned components、features、architecture connections、structure-watch signals の source of truth にします。そこから `monitor/application-monitor.html` と `monitor/application-monitor.ja.html` を生成します。

What The Owner Needs To Approve

  • structured product source data は、隠しフォルダではなく見える root の `product/` に置く。
  • application monitor を guardrail monitor / review monitor とは別の monitor として扱う。
  • `apps/api` と `apps/web` は、directory を作る前から planned component として追跡する。
  • 将来、実装 directory が application map と component contract に載らずに増えた場合、check を fail させる。
  • monitor HTML は source から生成し、手で編集しない。

Recommendation

app skeleton を作る前の baseline として承認することを推奨します。構造のズレを、直すコストが低いうちに見えるようにできます。

Risks And Mitigations

RiskMitigation
map が古くなる。`check-application-monitor.ps1` が生成 HTML の鮮度を確認し、map に載っていない実装 directory を検出する。
planned component が実装済み code と誤解される。monitor に component status を表示し、directory が存在した後に component contract を必須にする。
product monitor が guardrails と重複する。source と問いが違う。application structure は `product/application-map.json`、controls は `monitor/registry.json` から見る。

Runtime Boundary Check

application map には planned な FastAPI / Python と React / Vite / TypeScript components を記録するが、この変更では `apps/api`、`apps/web`、Docker images、dependency installs、runtime commands は作らない。
