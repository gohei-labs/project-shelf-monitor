Application Development Rules

Rules for how this application is allowed to evolve.

Development rules

APP-RULE-001

Application monitor source first

Changes to application requirements, development rules, components, or connections must update product/application-monitor.json before generated monitor HTML.

Evidence: check-application-monitor.ps1 verifies generated monitor freshness.

Active

Control: Hard gate

APP-RULE-002

Implementation directories must be declared

Every implementation directory under apps/, packages/, or services/ must be listed in the application monitor source and in project-components.json.

Evidence: The guardrail compares discovered implementation directories with the monitor source and the component contract.

Active

Control: Hard gate

APP-RULE-003

Requirement traceability before implementation

Product behavior should trace from requirement to feature and responsible component before implementation work begins.

Evidence: The requirements and architecture pages show requirement, feature, and component links separately.

Active

Control: Working agreement

APP-RULE-004

Phase 1 remains read-only

Phase 1 must not add controls or API behavior that moves, deletes, copies, imports, or rewrites owner files.

Evidence: Read-only behavior is tracked as an application development rule and a product requirement.

Active

Control: Working agreement

APP-RULE-005

Architecture changes get their own monitor page

Component, folder, stack, and connection decisions belong on the architecture page, not mixed into the rules or requirements page.

Evidence: The generator emits a dedicated architecture page.

Active

Control: Working agreement
