Substantive Record Check

This review asks for approval to require that a decision or change record actually have content before it satisfies the decide-before-implementing gate - closing the loophole where an empty, title-only file passed.

Requested Approval

  1. A record satisfies check-spec-before-impl only if its body has at least MinRecordChars (default 40) non-whitespace characters after stripping YAML frontmatter, HTML, and Markdown headings.
  2. The same substance check applies to a record linked from the PR body, not only to added or modified files.
  3. The failure message and -Json output distinguish a missing record from a thin one, naming the thin file.

What Changed

AreaRole after this change
scripts/check-spec-before-impl.ps1Adds Test-RecordHasSubstance and a MinRecordChars parameter; counts only substantive records as satisfying the gate.
Decision / change recordsAn empty or title-only changelog, ADR, or review file no longer passes the gate.

Risk

The threshold is a floor against emptiness, not a quality bar - a long but low-effort record could still pass, and owner review remains the real quality check. The floor is deliberately low (about one short sentence) to avoid rejecting a genuinely terse but real record; if a legitimate record is ever rejected, adding a sentence fixes it. Guardrail tooling stays excluded from the gate, so this change did not need a record of its own (one is provided anyway).

Terminology Check

TermMeaning
Decision recordA numbered ADR, change note, or review artifact that explains a change before it is implemented.
Substantive bodyThe record text that remains after removing frontmatter, HTML, and headings; it must clear a minimum length.
Thin recordA record file that matches the naming pattern but is empty or title-only - now rejected.
Floor, not a quality barThe check rejects emptiness; it does not judge whether the content is good. Owner review does that.

Runtime Boundary Check

This change is to a guardrail script only; it adds no runtime dependency. The check runs wherever the suite already runs: the local PowerShell pre-check, the CI check job, and the Dev Container Bash lane (which calls the same check.ps1). The authoritative environment remains the Dev Container plus Docker Compose.
