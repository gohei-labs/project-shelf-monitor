Guardrail Deep Dive Context Lock

文脈を固定する

LLM駆動開発では、AIに何度も同じ背景を説明するのではなく、プロジェクト内の文書に「判断の前提」を固定します。これにより、別のAIエージェント、人間、未来の自分が同じ文脈から作業を再開できます。

目的判断基準をチャットからプロジェクト資産へ移す。
防ぐ失敗AIが過去の決定を忘れ、同じ議論や危険な変更を繰り返すこと。
確認方法spec、ADR、change note、review artifact に残っているかを見る。
Meaning

何を「固定」するのか

固定するのは、単なるメモではありません。実装やレビューで判断に使う前提です。

方針

例: AIエージェント前提で開発する。旧フォルダは直接触らない。後回し作業はIssue化する。

境界

例: managed root、read-only phase、secrets、path validation、外部公開の扱い。

理由

例: なぜ Dev Container を選んだか。なぜ Copilot review を標準にしないか。

完了条件

例: check、CI、review record、deferred Issue が揃ってから PR ready と呼ぶ。

禁止事項

例: secrets を commit しない。chat だけに deferred work を残さない。

引き継ぎ情報

例: 何を変え、なぜ変え、どう検証し、何が残っているか。

Why It Matters

固定しないと何が起きるか

AIが古い前提で動く

過去に却下した選択肢や危険な作業を、別のturnや別agentがもう一度提案する。

人間がレビューできない

なぜその判断になったかがチャットに散らばり、承認対象が見えなくなる。

プロジェクトが資産化しない

LLM駆動開発の知見が会話の中に消え、次のプロジェクトで再利用できない。

Where It Lives

文脈の置き場所

文脈は種類ごとに置き場所を分けます。これが混ざると、あとから探せなくなります。

置き場所 何を固定するか このプロジェクトでの例 AIに期待する読み方
Spec 守るべき安定ルール docs/specs/ja/development-guardrails.md 実装前に該当ルールを読む。
ADR 大きな設計判断と理由 docs/ja/adr/0003-ai-agent-first-development.md 別案を出す前に、既存判断の理由を確認する。
Change note そのPRで何を変え、どう検証したか docs/changes/0004-guardrails-map.md PRの作業履歴と残リスクを読む。
Review artifact 人間が承認しやすい可視化 docs/ja/reviews/0003-guardrails-map.ja.html owner判断の対象と背景を確認する。
GitHub Issue 未完了の作業、リスク、後回し項目 Issue #8: guardrails map publishing 後回し作業を chat ではなく backlog として扱う。
Workflow

文脈を固定する流れ

1

会話で決まる

方向性、禁止事項、判断基準が出てくる。

2

種類を選ぶ

spec、ADR、change note、Issue のどれに置くか決める。

3

文書に昇格

チャットの要点を、後から読める形に書き直す。

4

PRで見せる

変更理由、検証、残リスクと一緒にレビューできる状態にする。

5

次の作業が読む

未来のAI/人間がその文脈を前提に作業する。

Anti-Patterns

固定したことにならないもの

弱い残し方

Avoid
  • チャットにだけ書く。
  • final summary にだけ書く。
  • PR comment にだけ書く。
  • ファイル名や場所が決まっていないメモに書く。
  • 英語の直訳で、owner がレビューできない文章にする。

強い残し方

Use
  • 安定ルールは spec に入れる。
  • 設計判断は ADR に入れる。
  • 実装単位の履歴は change note に入れる。
  • 後回しは acceptance criteria 付きの Issue にする。
  • owner 判断は HTML review artifact にする。
Owner Check

人間はここを見ればよい

細かい技術実装ではなく、「この文脈はちゃんと固定されているか」を見ます。

1. 置き場所は合っているか

ルールなのか、設計判断なのか、作業履歴なのか、後回しIssueなのか。

2. 次のAIが読めるか

実装前に参照すれば、同じ判断に戻れる粒度になっているか。

3. 自分が承認できるか

日本語で読んだとき、何を守るべきか、何が危険かが分かるか。

Terminology Check

This migrated review artifact keeps its original wording. Any future substantive edit should use standard documentation and engineering terminology.
