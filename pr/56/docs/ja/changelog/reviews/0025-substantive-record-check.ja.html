記録の実体チェック

この成果物は、決定／変更記録が「実装前に決める」ゲートを満たす前に、実際に中身を持つことを要求する変更の承認を求めます。空でタイトルだけのファイルが通っていた抜け穴を塞ぎます。

承認のお願い（Requested Approval）

  1. 記録が check-spec-before-impl を満たすのは、YAML フロントマター・HTML・Markdown 見出しを除去した本文が、少なくとも MinRecordChars（既定 40）の非空白文字を持つ場合のみ。
  2. 同じ実体チェックを、追加・変更ファイルだけでなく、PR 本文からリンクされた記録にも適用する。
  3. 失敗メッセージと -Json 出力は、記録が「無い」のか「薄い」のかを区別し、薄いファイルを明示する。

変更内容（What Changed）

対象変更後の役割
scripts/check-spec-before-impl.ps1Test-RecordHasSubstance と MinRecordChars パラメータを追加。実体のある記録のみゲートを満たすとみなす。
決定／変更記録空またはタイトルだけの changelog／ADR／レビューファイルはゲートを通らなくなる。

リスク（Risk）

閾値は「空でない」ための下限であって品質の基準ではない。中身の薄い長文なら通り得るし、品質の最終確認はオーナーレビューのまま。下限は意図的に低く（短い一文程度）し、実体はあるが極端に短い記録を弾かないようにしている。万一正当な記録が弾かれても、一文足せば解消する。ガードレールツールはゲートから除外のままなので、本変更自体には記録は不要（ただし用意している）。

用語チェック（Terminology Check / 標準用語チェック）

用語意味
決定記録（Decision record）変更を実装する前に説明する、番号付き ADR・変更ノート・レビュー成果物。
実体のある本文フロントマター・HTML・見出しを除いた後に残る記録テキスト。最小長を満たす必要がある。
薄い記録（Thin record）命名パターンには合うが、空またはタイトルだけのファイル。今回から却下される。
下限であって品質基準ではないこのチェックは空を却下するだけで、中身の良し悪しは判定しない。それはオーナーレビューの役目。

ランタイム境界チェック（Runtime Boundary Check）

本変更はガードレールスクリプトのみの変更で、新たなランタイム依存は無い。チェックはスイートが既に走る場所で走る：ローカルの PowerShell 事前チェック、CI の check ジョブ、Dev Container の Bash レーン（同じ check.ps1 を呼ぶ）。権威ある環境は引き続き Dev Container と Docker Compose。
