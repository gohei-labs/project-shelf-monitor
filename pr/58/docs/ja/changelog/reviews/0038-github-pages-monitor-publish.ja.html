GitHub Pages でのモニター公開

推奨CIモニター

判断サマリー

repository sources から monitor site を build し、main への push から deploy する GitHub Pages workflow を追加する。

オーナーに承認してほしいこと

GitHub Pages を、いつでも見られる人間向け monitor surface として承認する。source of truth は monitor/registry.json と review artifacts に残す。

推奨

進めてよい。オーナーは監視用の安定 URL を持てる一方で、CI は source から build し、Pages output を editable truth として扱わない。

リスクと軽減策

リスク軽減策
Pages が古い monitor output を deploy する。Pages workflow が scripts/generate/build-pages-site.ps1 を実行し、artifact package 前に monitor pages を再生成する。
Pages が第二の source of truth になる。artifact は artifacts/pages-site に build し、直接編集も commit もしない。
private repository content が Pages で露出する。artifact は repository 全体ではなく、monitor/ の HTML/registry data と HTML review artifacts だけをコピーする。
repository Pages settings が有効化されていない。URL に頼る前に、repository settings で Pages source を GitHub Actions に設定する。

Terminology Check

用語ここでの使い方
GitHub Pagesowner-facing monitor surface として使う hosted static site。
Pages artifactActions が deploy 用に upload する generated static files。
Source of truthmonitor を生成する編集対象の repository source。Pages output ではない。

Runtime Boundary Check

runtime は GitHub Actions と PowerShell のまま。Node、package manager、追加 build dependency は導入しない。
