設定可能なガードレール

この成果物は、各ガードレールをレジストリの enabled フィールドでプロジェクトごとに on/off できるようにし、プレイブックを他リポジトリへきれいに移植できるようにすることの承認を求めます。ただし「無効化は可視で理由付きの判断であり、黙った迂回ではない」という原則は保ちます。

承認のお願い（Requested Approval）

  1. playbook/registry.json の各ゲートに enabled（既定 true）を追加。enabled: false にする場合は空でない disabledReason を必須とする。
  2. scripts/check.ps1 がレジストリを読み、無効ゲートをスキップする。レジストリが読めない場合は全ゲートを実行（fail-closed＝エラー時に飛ばさない）。
  3. scripts/check-registry.ps1 がトグルと理由を検証し、地図は無効ゲートをグレー＋バッジ＋件数で表示。トグルはコミット済みレジストリ上でのみ行い、ブラウザ上の切替スイッチは無い。

変更内容（What Changed）

対象変更後の役割
playbook/registry.jsonon/off のスイッチ盤。各ゲートが enabled（無効時は disabledReason）を持つ。

リスク（Risk）

明白なリスクは、トグルが迂回手段になること（「ゲートを切ればいい」）。これは 3 点で抑える：無効化には記録された disabledReason が必須（強制）、変更はコミット済みレジストリと git レビューで可視、地図は無効ゲートを件数付きで明示。現状の制限：CI の別ジョブ human-review-artifact はゲートを check.ps1 経由でなく直接実行するため、それらを無効化してもまだ当該ジョブは止まらない。今日は全ゲートが有効なので挙動は不変。

用語チェック（Terminology Check / 標準用語チェック）

用語意味
enabled トグルこのリポジトリでガードレールを on/off するプロジェクト単位の真偽値（enabled）。
fail-closedエラー時（ここではレジストリが読めない時）は、飛ばさず全ゲートを実行するのが安全側の既定。
単一の正（Single source of truth）どのゲートが走るかはレジストリだけが決め、チェック実行器と地図が共にそれを読む。
無効理由（disabled reason）ガードレールを切る際に必須で記録される根拠。レジストリと地図に可視で残す。

ランタイム境界チェック（Runtime Boundary Check）

本変更はガードレール・ツールの変更であり、プロダクトではない。設定フィールドを 1 つ足し、PowerShell のチェック実行器と地図生成器にそれを読ませるだけ。新たなランタイムは導入しない：ローカル事前チェック（PowerShell）、CI の check ジョブ、Dev Container の Bash レーンは、いずれも同じ scripts/check.ps1 を実行するのでトグルを尊重する。

オーナー認識のための境界注記：CI の human-review-artifact ジョブはレビュー成果物ゲートを check.ps1 経由でなく直接呼ぶため、まだレジストリのトグルを読まない。権威ある環境は引き続き Dev Container と Docker Compose。
