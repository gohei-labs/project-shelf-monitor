Docusaurus ドキュメントポータル

Issue #16 のレビュー資料です。LLM が管理するルールや作業状態を、オーナーがブラウザで直接確認できる入口を作る判断です。

承認してほしいこと

  1. docs-portal/ を Docusaurus のソース置き場として採用する。
  2. 最初のブラウザ管理画面として、LLM 作業ガードレール一覧を日本語・英語で用意する。
  3. CI で Docusaurus の build を確認し、壊れたポータルをマージしない。
  4. LLM-GR-004 を Active に変更し、ブラウザで見える管理画面を今後の標準ルールにする。

今回作るもの

ポータル本体

docs-portal/ に Docusaurus の最小構成を置きます。ルートの未追跡 package.json は使いません。

ガードレール画面

LLM 作業ガードレールの状態を、ブラウザで確認できるページとして作ります。

CI チェック

GitHub Actions で Node.js を用意し、ポータルの build をチェックします。

必読文脈ゲート

docs/agent-core-context.json に、再開後や別エージェントへの引き継ぎ時に読む固定パックを定義します。

技術選定

候補判断理由
Docusaurus採用Markdown と React/MDX を両立でき、静的ドキュメントサイトとして拡張しやすい。
VitePress今回は見送り軽量だが Vue 前提が強く、今回の React 寄りの将来像とは少し離れる。
Material for MkDocs今回は見送り読みやすいドキュメントには強いが、将来の React UI 連携は Docusaurus の方が自然。
Astro Starlight今回は見送り静的サイトとして魅力的だが、今回の管理ポータル第一歩では Docusaurus を優先する。

今回やらないこと

  • 外部公開の本番デプロイはまだ行いません。
  • 既存 Markdown の全ページ移植はまだ行いません。
  • Windows ホストへの npm 導入は前提にしません。Docusaurus build は CI と Docker / Dev Container 側で担保します。
  • 既存日本語 Markdown の文字化け全面修正は、このレビュー中のオーナー指摘を受けて Issue #37 で追跡します。今後の後回し作業は、Issue 作成前にオーナー確認を必須にします。

実行環境境界チェック

この PR で見つかったミスは、Windows 側で npm が見つからないという局所的な事実に引っ張られ、既存の開発環境方針を再確認しないまま「Windows npm 整備」を別作業のように書いたことです。

境界今回の判断
Windows ホスト操作端末です。プロジェクトの Node/npm 実行環境の正本にはしません。
Docker / Dev Container再現可能なローカル実行経路です。ローカルで build する場合はここに寄せます。
CIDocusaurus build を必ず検証する正本のチェック経路です。
ガードレール今後、レビュー資料で実行環境やホスト環境に触れる場合は、この Runtime Boundary Check を必須にします。

標準用語チェック

使う用語意味
Documentation portalドキュメントをブラウザで確認するための静的サイト。
Control surface人間が現在のルールや状態を確認し、統制を保つための画面。
Static site generationビルド時に HTML などの静的ファイルを生成し、ホスティングする方式。
Runtime boundaryWindows ホスト、Docker / Dev Container、CI のどこを正本の実行環境にするかという境界。
Deferred work owner confirmation後回し作業を見つけた時点で、Issue 化するかどうかをレビュー依頼前にオーナーへ確認すること。
Core context引き継ぎ、再開、コンテキスト圧縮後に、AI エージェントが最初に読む固定の必読パック。

引き継ぎ強化

この PR では、固定の Core Context 読み込みゲートを追加します。目的は、あらゆるキーワードを検知することではありません。コード、ドキュメント、ワークフロー、GitHub 状態、レビュー記録を変更する前に、どのエージェントも同じ最低限の文脈から始めることです。

層追加する統制
正本docs/agent-core-context.json に always-read と条件付きで読むファイルを定義します。
人間向け説明docs/agent-core-context.md に、この読み込みゲートをいつ適用するかを書きます。
ローカルと CIscripts/check-agent-core-context.ps1 で manifest が壊れていないか確認し、標準チェックに含めます。
PR 運用PR テンプレートで Core Context を読んだか確認します。
