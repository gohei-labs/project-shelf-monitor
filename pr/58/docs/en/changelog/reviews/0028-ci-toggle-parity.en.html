Registry Toggle Parity in CI

This review asks for approval of a fix for the gap called out in review 0024: CI's human-review-artifact job ran its four review gates directly, so the registry enabled toggle did not reach them. Disabling one of those guardrails was silently ignored on CI, and a recorded disabledReason gave false assurance. The fix makes every enforcement lane honor the same toggle.

Requested Approval

  1. The four CI-invoked gates now consult the registry and skip when their guardrail is disabled, so the registry is the single on/off switch for them too.
  2. review-artifact-runtime-boundary ran in two lanes with split-brain behavior when disabled; both lanes now agree.
  3. check-registry.ps1 now fails if a CI-wired script gate forgets the toggle, so the bypass cannot silently return.

What Changed

AreaRole after this change
scripts/guardrail-lib.ps1New shared helper: one Test-GuardrailEnabled that reads the registry, fail-closed.
scripts/check.ps1Sources the helper instead of its own copy - one toggle implementation.
The four review gatesDot-source the helper and exit early when disabled, so CI honors the toggle.
scripts/check-registry.ps1Asserts every CI-wired script gate consults the toggle.
scripts/check-spec-before-impl.ps1Excludes guardrail-lib.ps1 as guardrail tooling.

Risk

The change is to the guardrail tooling, not the product. With every guardrail enabled today, behavior is unchanged. The one remaining limit is the devcontainer-build guardrail: it is a CI job with no gate script (it builds the Dev Container), so its job-level toggle is still not wired and the new assertion - which covers script gates only - does not flag it. That is tracked as a separate follow-up.

Terminology Check

TermMeaning
Enforcement laneA path by which a guardrail actually runs: the local pre-check, the CI check job, the CI human-review-artifact job, or the Dev Container Bash lane.
Toggle bypassAn enforcement lane that runs a gate without reading the registry enabled flag, so a disabled guardrail still runs there.
Split-brainTwo lanes disagreeing about whether a guardrail is on - here, the check job skipping a disabled gate while CI kept enforcing it.
Fail-closedOn an error (an unreadable registry) the safe default is to treat every guardrail as enabled, not to skip.

Runtime Boundary Check

No new runtime is introduced. The change adds one PowerShell helper file and teaches the existing gate scripts to read the registry. The local pre-check (PowerShell), the CI check job, and the Dev Container Bash lane all run the same scripts/check.ps1; the CI human-review-artifact job runs the four gate scripts directly with Node-free PowerShell. After this change all of those lanes consult the same registry toggle. No new dependency (Node, npm, Docker, package manager) is added.
