GitHub Pages Monitor Publish

RecommendedCIMonitor

Decision Summary

Add a GitHub Pages workflow that builds the monitor site from repository sources and deploys it from pushes to main.

What The Owner Needs To Approve

Approve GitHub Pages as the always-available human monitor surface, with monitor/registry.json and review artifacts remaining the source of truth.

Recommendation

Proceed. The owner gets a stable URL for monitoring, while CI still builds from source and avoids treating Pages output as editable truth.

Risks And Mitigations

RiskMitigation
Pages deploys stale monitor output.The Pages workflow runs scripts/generate/build-pages-site.ps1, which regenerates monitor pages before packaging the artifact.
Pages becomes a second source of truth.The artifact is built into artifacts/pages-site and is not edited directly or committed.
Private repository content is exposed by Pages.The artifact copies only monitor/ HTML/registry data and HTML review artifacts, not the whole repository.
Repository Pages settings are not enabled.Set Pages source to GitHub Actions in repository settings before relying on the URL.

Terminology Check

TermUse here
GitHub PagesThe hosted static site used as the owner-facing monitor surface.
Pages artifactThe generated static files uploaded by Actions for deployment.
Source of truthThe editable repository source used to generate the monitor; not the Pages output.

Runtime Boundary Check

The runtime remains GitHub Actions plus PowerShell. No Node, package manager, or extra build dependency is introduced.
