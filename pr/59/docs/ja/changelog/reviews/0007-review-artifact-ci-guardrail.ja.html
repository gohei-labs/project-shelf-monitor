Human Review Issue #21 CI Guardrail

HTMLレビュー資料の作成漏れをCIで止める

PRがオーナー承認や人間レビューを求めているのに、ブラウザで読めるHTML資料がない状態を、レビュアーの注意力ではなくCIで検出します。 これはレビュー品質の最低ラインをシステム側に寄せるためのガードレールです。

承認してほしいこと
  1. 人間レビューが必要なPRでは、docs/reviews/*.html へのリンクを必須にする。
  2. リンク先HTMLがPR branchに存在しなければCIで失敗させる。
  3. specs / workflows / PR template / Dev Container などの高リスク変更は、人間レビュー対象として扱う。
Problem

なぜ必要か

PR #20 で、レビュー依頼は出ていたのにHTML review artifactが後から追加されました。この種類の漏れは、レビュー担当者が気づくかどうかに依存させるべきではありません。

Failure

承認対象が見えない

PR本文だけでは、オーナーが何を判断すべきか見落としやすくなります。

Review Risk

レビュアー依存になる

「HTMLがない」と人が気づけなければ、そのまま進んでしまいます。

Guardrail

CIで止める

必要なHTMLがなければ、PRのcheckが失敗するようにします。

Detection

CIが見るもの

完璧な意味理解ではなく、実用的な検出条件を組み合わせます。

検出対象 条件 結果
PR本文 Requested Review Decision、承認してほしいこと、Owner Approval などの語がある。 HTML必須
高リスク変更 docs/specs/、.github/workflows/、.github/pull_request_template.md、.devcontainer/、ADRなどを変更する。 HTML必須
HTMLリンク PR本文に docs/reviews/*.html が含まれる。 存在確認
存在しないHTML PR本文に書かれたHTMLがbranchに存在しない。 CI失敗
Flow

PR作成時の流れ

1

PRを開く

PR本文と変更ファイルをCIが読みます。

2

必要性を判定

承認依頼または高リスク変更ならHTMLが必要です。

3

HTMLを確認

PR本文の docs/reviews/*.html が存在するか見ます。

4

通す / 止める

足りなければCI失敗。足りていれば次のレビューに進みます。

Limits

このガードレールの限界

これは必要条件を強制する仕組みであって、レビュー品質そのものを完全保証するものではありません。

できないこと
  • HTMLの内容が分かりやすいかまでは判定しない。
  • 人間の最終判断を置き換えない。
  • すべてのPRのリスクを完璧に分類しない。
できること
  • HTMLの作成漏れをCIで止める。
  • 存在しないHTMLリンクをCIで止める。
  • specsやCI変更などを、人間レビュー対象として扱う。
Approval Checklist

レビュー時に見るポイント

確認 見ること 今回の提案
強制力 レビュー担当者の注意力ではなくCIで止めるか。 はい
範囲 すべてのPRではなく、承認依頼と高リスク変更に絞っているか。 はい
透明性 何が足りないかをCI outputで説明できるか。 はい
残るリスク HTMLの品質まではCIでは保証できないことを認識しているか。 人間レビューで補う
