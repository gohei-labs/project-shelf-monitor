Split reusable and project-specific rules into a Project rules view

This review asks for approval to add a dedicated "Project rules" monitor view that surfaces the rules tagged scope: product, and keep the Guardrails view for reusable rules only.

Requested Approval

  1. Add a Project rules view. A new generated page (monitor/project-rules.html + .ja.html) lists the guardrails whose registry scope is product — currently 6: Dev Container build, Implementation component contract, Backend lint and tests, Frontend build, Compose config validation, Application monitor freshness. The page is driven by the scope field, not a hardcoded list, so re-tagging a rule moves it between views with no code change.
  2. Keep Guardrails reusable-only. The Guardrails view now shows the 21 reusable rules (scope generic + bilingual) and no longer lists the project-specific ones.
  3. Add the Project rules tab to the monitor nav on every page (between Guardrails and Development rules), in both English and Japanese.

Risk

The split is presentation only: both views read the same registry and the same scope field, so no rule changes how it is enforced. The registry consistency guardrail (build-guardrail-monitor.ps1 -Verify) covers the new pages, so they cannot drift from the registry. pwsh scripts/check.ps1 passes.

Not Decided Here

  • Whether the current 6 scope: product tags are the right set — re-auditing which existing rules are project-specific is a separate follow-up task.
  • The original four-category taxonomy (generic / stack / domain / owner-environment) is not introduced; scope stays generic vs product for now.
  • No rule text, gate, or enforcement is rewritten.
