Pages Plan Fallback

RecommendedCIPagesToken boundary

Decision Summary

Keep the monitor workflow green when GitHub Pages is unavailable by uploading the monitor as a normal Actions artifact and skipping only the Pages deployment.

What The Owner Needs To Approve

Approve artifact fallback as the controlled behavior for repositories where GitHub Pages is not enabled or not supported by the current account plan.

Recommendation

Proceed. The token now has repository admin access, but the Pages API reports that the current plan does not support Pages for this private repository. The workflow should reflect that external constraint instead of staying red.

Risks And Mitigations

RiskMitigation
Main Pages workflow fails even though the monitor built correctly.Deploy job checks Pages availability and skips deployment when Pages is unavailable.
Owner loses access to the latest monitor.Main runs upload guardrail-monitor-main-<run_id> as a normal Actions artifact.
Token permissions are confused with plan support.Token-management docs now separate Pages token scopes from repository plan/visibility support.
A skipped deploy hides a real build failure.The build job and guardrail checks still fail normally; only unavailable Pages deployment is skipped.

What Is Not Decided Here

This does not upgrade the GitHub plan, change repository visibility, or choose another hosting provider. It only keeps the monitor artifact available while those choices remain external to the workflow.

Terminology Check

Runtime Boundary Check

The runtime remains GitHub Actions plus PowerShell. The change adds a REST preflight and artifact upload behavior; it does not add a new runtime, package manager, or deployment provider.
