Bash Lane Guardrail Parity

This review asks for approval to make the Dev Container / CI Bash lane enforce the full guardrail suite by delegating scripts/check.sh to PowerShell, and to remove the divergent Bash copies of two gates.

Requested Approval

  1. scripts/check.sh delegates to pwsh scripts/check.ps1 as the single source of truth for the Bash lane, instead of running a hand-maintained subset of gates.
  2. The Dev Container gains PowerShell 7 via ghcr.io/devcontainers/features/powershell so the delegation works in the container and in the devcontainer CI job.
  3. The orphaned scripts/check-agent-core-context.sh and scripts/check-docs-portal.sh are removed; their PowerShell twins already run inside check.ps1.
  4. The Bash lane fails closed when pwsh is absent, rather than passing with no checks run.

Risk

Before this change the devcontainer CI job ran only two guardrail scripts (core-context and docs-portal), so a failure in any of the other gates could still pass there. After it, the Bash lane runs the entire scripts/check.ps1 suite - the same gates the Windows host and the CI check job run, including text encoding, bilingual pairs, language roots and content, document naming, registry integrity, secret-leak, decision-before-implementation, and the review-artifact runtime-boundary check. The four PR-context review gates and the devcontainer-build job stay in their own CI jobs and are unchanged. The main residual cost is that the devcontainer job builds the docs portal twice (quick then full); that is a performance cost, not an enforcement gap.

Terminology Check

TermMeaning
GuardrailA rule enforced by a script that fails the build, not by a prompt.
Check laneOne environment in which the guardrail suite runs: Windows host, PowerShell CI job, or Dev Container Bash.
Single source of truthOne authoritative implementation (scripts/check.ps1) that all lanes call, so they cannot drift.
Fail closedWhen the tool cannot run (here, pwsh missing), the check fails rather than passing.

Runtime Boundary Check

This change is about the runtime boundary itself. The authoritative environment remains the Dev Container plus Docker Compose, not whichever shell a host happens to have. PowerShell 7 (pwsh) is now a declared Dev Container dependency, installed through the official PowerShell feature rather than assumed on the host.

On a Windows host the suite already runs through pwsh scripts/check.ps1. On Linux, macOS, or CI it runs through bash scripts/check.sh, which now execs that same PowerShell suite. No Windows-only tooling is introduced. Node and npm - already provided by the Node Dev Container feature - remain required for the docs-portal build.
