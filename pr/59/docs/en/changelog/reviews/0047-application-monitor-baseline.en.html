Application Monitor Baseline

This change adds a separate monitor for application development before application code is introduced.

Decision Summary

Create `product/application-monitor.json` as the source of truth for application development rules, requirements, features, and architecture assumptions. Make the default application monitor page the rules page, with separate requirement, feature, and architecture surfaces.

What The Owner Needs To Approve

  • Use visible root `product/` for structured application monitor source data, not a hidden folder.
  • Treat the application monitor as separate from the guardrail and review monitors.
  • Show guardrails, reviews, development rules, requirements, features, and architecture as sibling top-navigation items.
  • Make application-specific development rules the default application monitor page.
  • Do not add a low-information table-of-contents page before the rules.
  • Keep requirements, features, and architecture/component decisions on separate pages.
  • Use compact cards plus detail modals for development rules, requirements, and features.
  • Render architecture as a generated diagram, not as component and connection lists.
  • Keep progress and component-contract status out of the architecture diagram nodes.
  • Keep progress metadata, source links, and managed-root summaries out of these monitor pages.
  • Fail future checks if implementation directories appear without monitor-source and component-contract coverage.
  • Keep the monitor generated from source; do not hand-edit monitor HTML.

Recommendation

Approve this baseline before creating the app skeleton. It makes development rules, requirements, features, and the architecture diagram reviewable before code starts to grow.

Risks And Mitigations

RiskMitigation
The monitor source can become stale.`check-application-monitor.ps1` verifies generated HTML and flags implementation directories missing from the architecture section.
Planned components may be mistaken for implemented code.The monitor shows component status and the guardrail requires component contracts only after directories exist.
The application monitor duplicates guardrails.The source and question are different: application development rules come from `product/application-monitor.json`, while general controls come from `monitor/registry.json`.

Runtime Boundary Check

The architecture diagram shows planned FastAPI / Python and React / Vite / TypeScript components, but this change does not create `apps/api`, `apps/web`, Docker images, dependency installs, or runtime commands.
