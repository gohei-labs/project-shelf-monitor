レビュー資料の staged 検出

Project Shelf レビュー資料。ステータス: オーナーレビュー用ドラフト。日付: 2026-06-14。

英語版: 0013-review-artifact-staged-detection.en.html

判断の概要

推奨明示的な changed-file list が渡されない場合、ローカルのレビュー資料ガードレールは staged changes も確認するようにします。

これにより、AIエージェントが新しいレビューHTMLを stage した後にローカルチェックを実行しても、誤って none と表示される問題を防ぎます。

承認してほしいこと

  • ChangedFiles が省略された場合、ローカルスクリプトは Git の working-tree diff と staged diff を読んでよい。
  • stage された新規 docs/reviews/*.html は、用語チェックと参考資料チェックのガードレールで必ず検査する。
  • 関係のないローカル未追跡ファイルを巻き込まないため、untracked files は自動検出に含めない。

ここでは決めないこと

  • documentation portal 全体の実装。
  • PowerShell helper の共通化。
  • 今回と無関係な既存ファイルの line-ending normalization。

変更前と変更後

場面 変更前 変更後
CIのPRチェック GitHub Actions から渡される明示的な changed files を使う。 変更なしこれまで通り、明示的な changed files を使う。
新規レビューHTMLを stage したローカル実行 問題あり変更されたレビュー資料があるのに none と表示されることがあった。 修正git diff --cached --name-only を読み、stage 済みのレビュー資料を検査する。
関係ない未追跡ファイルがあるローカル実行 検査対象に含めていなかった。 意図通り今後も自動検出には含めない。

検出の流れ

エージェントがファイルを stage
->
ローカルガードレール
->
git diff
git diff --cached
->
レビューHTMLを検査

リスクと対応

リスク 対応
過信untracked files はローカルチェックで見落とされる可能性が残る。 これは意図した仕様です。エージェントは、PRに含めるつもりのファイルを stage してからローカルチェックに頼るべきです。
重複3つのスクリプトに似た Git changed-file logic が入る。 今回は変更を小さく保ちます。挙動が増えた場合だけ、後で shared helper へ切り出します。

用語チェック

用語 意味
staged changes git add によって Git index に追加済みの変更。
working-tree diff まだ stage されていない tracked file の変更。
untracked files まだ Git で追跡されていないファイル。ローカル自動検出からは意図的に除外する。
guardrail 既知の作業ミスが再発しないようにする、繰り返し実行できるチェック。
