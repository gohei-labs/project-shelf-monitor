Human Review Issue #13 LLM Work Guardrails

LLM作業全般のガードレール台帳を作る

これは開発ルールブック専用ではありません。LLMに作業を任せるときに繰り返し出るズレを、 軽量な台帳として管理する提案です。追加、保留、廃止を簡単にできることを重視します。

Approve / Revise

判断してほしいこと

  1. LLM work guardrails を独立した台帳として管理してよいか。
  2. 初期3件を `Proposed` として登録してよいか。
  3. 状態を `Proposed / Active / Paused / Deprecated` の4つで管理してよいか。
Purpose

何を解決するか

LLM作業で繰り返し起きる「言葉のズレ」「省略の見えにくさ」「資料目的の混在」を、次回以降の作業前に参照できる形にします。

Lightweight

重くしない

長いチェックリストではなく、繰り返し効く少数のguardrailsだけを置きます。

Editable

後から変えられる

追加、保留、廃止、削除を前提にします。固定された憲法にしません。

General

作業全般に効く

codingだけでなく、調査、資料作成、レビュー、説明にも適用します。

Lifecycle

4つの状態

ルールは強制するだけではなく、試す、止める、廃止する、を明示できるようにします。

Proposed

候補

owner feedbackやreview findingから出た候補。まだ標準ルールではありません。

Active

有効

今後のLLM-assisted workで標準的に適用します。

Paused

保留

邪魔、曖昧、逆効果になった場合に一時停止します。

Deprecated

廃止

今後は使わないが、なぜ使わなくなったかの履歴を残します。

Human Quick Check

人間はここだけ先に見る

全体台帳を毎回読む必要はありません。オーナーレビューでは、まずこの4点だけ確認します。

確認見ること今回の状態
用語 重要語が、他の人や将来のAIエージェントにも伝わる標準的な言葉になっているか。 Proposed 標準用語を優先する。
省略 省略があるなら、例、抜粋、代表例、省略あり、と明記されているか。 Proposed 省略を明示する。
目的 承認依頼、再利用可能資料、実装記録、背景説明のどれなのか分かるか。 Proposed 目的が違う成果物は分ける。
状態 各ガードレールが `Proposed / Active / Paused / Deprecated` のどれか分かるか。 Visible 初期3件はすべて `Proposed`。
Control Surface

管理画面としては別途ポータルが必要

このPRで作っている台帳は、LLMが参照する正本です。ただし、Markdownだけでは人間が常時チェックできる管理画面にはなりません。 オーナーがLLMに要約を頼まなくても、現在のルール、状態、レビュー記録、後回し作業をブラウザで確認できる画面が必要です。

層役割今回の扱い
Markdown 台帳 リポジトリ上の正本。AIエージェントと履歴管理に向く。 このPR `docs/llm-work-guardrails/registry.md` に追加。
HTML レビュー成果物 PR単位・判断単位で、人間が承認しやすくする画面。 このPR このレビューHTMLで提供。
Docs Portal 現在のルール、状態、Issue、PR、レビュー記録を常時確認する管理画面。 Issue #16 別タスクで技術選定と実装を行う。
