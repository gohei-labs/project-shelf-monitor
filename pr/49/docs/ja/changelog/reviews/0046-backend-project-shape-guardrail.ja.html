Backend Project Shape Guardrail

RecommendedGuardrailBackendCI

Decision Summary

backend conditional check を締め、`apps/api` があるのに `apps/api/pyproject.toml` が無い状態を許可しない。

What The Owner Needs To Approve

`apps/api` が存在した時点で、その backend project は必須チェック対象になる、というルールを承認する。backend はまだ無くてもよいが、lint と tests を skip できる半端な形では作れない。

Recommendation

進める。conditional guardrail は、対象が存在する前だけ inactive であるべき。backend path が現れたら、project shape と checks を強制する必要がある。

Risks And Mitigations

RiskMitigation
downstream project が別の backend path を使いたい。この template では `apps/api` を標準 path とし、変更する場合は guardrail/docs を明示的に更新する。
agent が早すぎる placeholder `apps/api` directory を作る。guardrail が即座に失敗し、placeholder を消すか、正式な Python project metadata を追加するよう促す。
このルールが、今すぐ backend を必須にすると誤解される。`apps/api` が存在しない間は引き続き skip する。backend path が存在した時だけ厳格化する。

What Is Not Changed

backend application code、API framework、dependencies は追加しない。app skeleton 作業前に backend を必須化する変更でもない。

Terminology Check

TermUse here
Conditional guardrail対象が存在する前だけ skip され、対象が現れたら blocking になる gate。
Project metadata再現可能な tooling に必要な Python project file である pyproject.toml。
Backend project shapeapps/api 配下に backend code を置く、期待される repository layout。

Runtime Boundary Check

runtime boundary は既存の guardrail suite 内に留まる。変更は scripts/check.ps1 を更新し、apps/api があるのに pyproject.toml が無い場合に CI と local checks を失敗させるもの。新しい runtime、package manager、server process、host-machine requirement は追加しない。
