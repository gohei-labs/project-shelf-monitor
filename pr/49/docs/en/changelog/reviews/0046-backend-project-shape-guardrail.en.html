Backend Project Shape Guardrail

RecommendedGuardrailBackendCI

Decision Summary

Tighten the backend conditional check so `apps/api` cannot exist without `apps/api/pyproject.toml`.

What The Owner Needs To Approve

Approve the rule that a backend project becomes mandatory-check territory as soon as `apps/api` exists. The backend may be absent, but it may not be half-created in a shape that skips lint and tests.

Recommendation

Proceed. A conditional guardrail should be inactive only before the target exists. Once the backend path exists, the project shape and checks must be enforced.

Risks And Mitigations

RiskMitigation
A downstream project wants a different backend path.Keep `apps/api` as the standard path for this template and require an explicit guardrail/docs update before changing it.
Agents create a placeholder `apps/api` directory too early.The guardrail fails immediately, forcing the agent to either remove the placeholder or add the real Python project metadata.
The rule is mistaken for requiring a backend today.The check still skips while `apps/api` is absent; it only becomes strict once the backend path exists.

What Is Not Changed

This does not create backend application code, choose an API framework, add dependencies, or require a backend before the app skeleton work begins.

Terminology Check

TermUse here
Conditional guardrailA gate that is skipped only before its target exists, then becomes blocking when the target appears.
Project metadatapyproject.toml, the Python project file needed for reproducible tooling.
Backend project shapeThe expected repository layout for backend code under apps/api.

Runtime Boundary Check

The runtime boundary stays within the existing guardrail suite. The change updates scripts/check.ps1 so CI and local checks fail when apps/api exists without pyproject.toml. It does not add a new runtime, package manager, server process, or host-machine requirement.
