ルールとモニターの分類整理

推奨GovernanceNaming

Decision Summary

live source-document layer の名前を playbook から rules に変え、生成された browser-visible layer の root playbook を monitor に変える。

What The Owner Needs To Approve

repository taxonomy を承認する。docs/ はルールの正本、monitor/ は生成された表示、scripts/ は guardrails の強制層。

Recommendation

進める。docs は六法全書のようにルールを保持し、monitor が表示の仕方を決める、というオーナーの整理と一致する。

Risks And Mitigations

RiskMitigation
rename 後に既存 links が壊れる。live docs、scripts、registry paths、indexes を更新し、monitor pages を再生成する。
historical records に古い名前が残る。古い記録は履歴として保持し、過去 state の説明だと明記する。
AI agents が古い required-reading manifest を読む。.agents/agent-core-context.json を docs/en/rules/ に更新する。

What Is Not Decided Here

monitor UI の再設計、新しい docs portal、将来の application architecture は決めない。repository の命名と責務境界だけを直す。

Terminology Check

TermUse here
Rulesdocs/{en,ja}/rules/ に置く恒久 source documents。
Monitormonitor/ 配下の generated browser-visible pages と machine-readable registry。
Scriptsscripts/ 配下の executable guardrail checks と generators。

Runtime Boundary Check

変わる runtime boundary は file path だけ。generator と check scripts が monitor/ を読む/書くようになる。新しい server、external service、secret、network dependency は追加しない。
