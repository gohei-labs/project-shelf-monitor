Text Encoding Guardrail Review

文字化けを「気をつける」ではなく、標準方針・スクリプト・CIで止めるためのレビュー資料。

Approve These Decisions

今回承認してほしいこと

  • リポジトリ内のテキストファイル標準を UTF-8 without BOM に固定する。
  • PowerShell 7 を、このリポジトリでの推奨シェルにする。ただし、このPRでは必須化しない。
  • Windows PowerShell 5.1 では、読み書き時の encoding 明示と、.ps1 内の非ASCIIリテラル回避を必須にする。
  • scripts/check-text-encoding.ps1 を標準チェックに組み込み、文字化けをCIで止める。
Preventable

避けられるもの

ファイル自体が UTF-8 ではない、BOM が混ざる、UTF-16で保存される、文字化け済みの文字列がコミットされる、という問題はチェックで止められます。

Display Only

残るもの

Git が日本語パスを \347... のように表示する、PowerShell 5.1 が -Encoding UTF8 なしで誤読する、といった表示上の問題は残ります。これは破損と区別します。

Stop In CI

止める条件

無効なUTF-8、UTF-8 BOM、UTF-16 BOM、典型的な文字化けマーカーが追跡対象テキストに入った場合、標準チェックを失敗させます。

原因と対策の対応表

原因 起きること 対策
Windows PowerShell 5.1 の既定読み込み BOMなしUTF-8の日本語が文字化けして見える。 Get-Content -Encoding UTF8 を必須にし、PowerShell 7 を推奨する。
Windows PowerShell 5.1 の既定書き込み Set-Content、Out-File、> で別の文字コードになる。 書き込み時に encoding を明示し、CIで非UTF-8を検出する。
$OutputEncoding が US-ASCII 外部コマンドへ渡す日本語が壊れる可能性がある。 日本語をパイプで渡す運用を避け、必要ならUTF-8を明示する。
Git の core.quotePath 日本語ファイル名がエスケープ表示される。 人間向け確認では git -c core.quotepath=false ... を使う。

導入するガードレール

1. Editor policy

.editorconfig で UTF-8、LF、final newline を指定。

2. Encoding check

scripts/check-text-encoding.ps1 で追跡対象テキストを検査。

3. Standard check

scripts/check.ps1 から encoding check を必ず実行。

4. Spec

英語版・日本語版の仕様に、安全な読み書きと診断方法を記録。

PowerShell 7

バージョンアップの扱い

PowerShell 7.6.2 は公式MSIでインストール済みです。C:\Program Files\PowerShell\7\pwsh.exe から起動でき、scripts/check.ps1 も PowerShell 7 で通っています。

ただし、既存のターミナルやエージェントプロセスは、再起動するまで更新後の PATH を拾わないことがあります。その間は絶対パスで pwsh.exe を呼びます。
