What To Watch

このプロジェクトで一番大事にすること

コードを書く速さより、変更が安全に進む道筋を作ることを優先します。AIエージェントは強い実装力を持つ一方で、文脈の取り違え、確認漏れ、後回し作業の消失、広すぎる変更を起こしやすいためです。

作業を小さくする

Scope

Issue、branch、PR の intent を1つに絞り、レビュー可能な大きさを保つ。

証拠を残す

Audit

CI、check results、Codex Review Record、Issue links で「やったこと」を追跡できるようにする。

安全側に倒す

Safety

file operation、secrets、external access、storage は read-only と明示的承認を基本にする。

Development Path

開発の道筋

AIエージェントが自由に動くのではなく、この順路に沿って進めます。各地点で次へ進む条件を置くことで、暴走や漏れを抑えます。

1

Issue

目的、acceptance criteria、risk、関連specを置く。

2

Branch

1つの intent に絞った short-lived branch で作業する。

3

Implement

編集前に周辺ファイルと関連specを読む。大きく作りすぎない。

4

Check

scripts/check を実行する。できないcheckは理由を書く。

5

Docs

behavior が変わるなら spec または change note を更新する。

6

PR

変更内容、検証、残リスク、後回しIssueを明記する。

7

AI Review

review-only pass を実行し、Codex Review Record を残す。

8

Merge

checks、review、deferred Issue の記録が揃ってから進める。

Enforcement

ガードレールの強さ

すべてのガードレールが同じ強さではありません。今の段階では「仕様」「スクリプト」「CI」「PRコメント」「Issue/Project」の組み合わせで段階的に強くします。

強さ 仕組み 今の使い方 次に強くする余地
Policy Specs / ADRs 判断基準を固定し、AIエージェントが読む前提を作る。 新しい機能領域ごとに spec を追加する。
Script scripts/check 人間、CI、AIが同じ validation entry point を使う。 app が増えたら backend/frontend/browser checks を実体化する。
CI GitHub Actions PR diff の hygiene と将来の test/build を記録する。 branch protection と required checks を有効化する。
Audit PR comments / Codex Review Record AI review が本当に行われた証拠を GitHub に残す。 review type ごとの template と checklist を増やす。
Tracking Issues / GitHub Project 後回し、risk、acceptance check を消さない。 Issue template と project automation を追加する。
Failure Modes

この資料が防ぎたい失敗

AIが勝手に広げる

対策: Issue intent、small PR、spec参照、PR size rule。

後回しが消える

対策: deferred work は必ず別Issue。PR commentだけでは不十分。

レビューしたことにする

対策: Codex Review Record を PR の top-level comment に残す。

ローカルファイルを危険に触る

対策: read-only first、path validation、dry-run、専用security review。

なぜそうしたか忘れる

対策: docs/specs、ADRs、docs/changes を更新する。

人間が承認できない

対策: HTML review artifact で説明と承認対象を分ける。
