Configurable Guardrails

This review asks for approval to make each guardrail turn-on/off-able per project via an enabled field in the registry, so the playbook ports cleanly to other repositories - while keeping the rule that disabling a guardrail is a visible, explained decision, not a silent bypass.

Requested Approval

  1. Add enabled (default true) to every guardrail in playbook/registry.json; setting enabled: false requires a non-empty disabledReason.
  2. scripts/check.ps1 reads the registry and skips disabled guardrails. If the registry cannot be read it runs every guardrail (fail-closed - never skip on error).
  3. scripts/check-registry.ps1 validates the toggle and reason; the map shows disabled guardrails greyed with a badge and a disabled count. Toggling lives only in the committed registry - there is no in-browser switch.

Risk

The obvious risk is that a toggle becomes a bypass - "just disable the gate". This is contained three ways: disabling requires a recorded disabledReason (enforced), the change is visible in the committed registry and in git review, and the map renders disabled guardrails distinctly with a count. One current limit: CI's separate human-review-artifact job runs its gates directly, not through check.ps1, so disabling those would not stop that job yet. With every guardrail enabled today, behavior is unchanged.

Terminology Check

TermMeaning
Enabled toggleA per-project boolean (enabled) that turns a guardrail on or off in this repository.
Fail-closedOn an error (here, an unreadable registry) the safe default is to run every guardrail, not to skip.
Single source of truthThe registry alone decides which guardrails run; the check runner and the map both read it.
Disabled reasonA required, recorded justification for turning a guardrail off, kept visible in the registry and the map.

Runtime Boundary Check

This change is to the guardrail tooling, not the product. It adds a config field and teaches the PowerShell check runner and map generator to read it. It introduces no new runtime: the local pre-check (PowerShell), the CI check job, and the Dev Container Bash lane all run the same scripts/check.ps1 and therefore honor the toggle.

One boundary note for owner awareness: CI's human-review-artifact job invokes its review-artifact gates directly rather than through check.ps1, so it does not yet read the registry toggle. The authoritative environment remains the Dev Container plus Docker Compose.
