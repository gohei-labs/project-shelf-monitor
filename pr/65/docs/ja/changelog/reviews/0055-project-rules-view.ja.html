再利用可能ルールとプロジェクト固有ルールを「Project rules」ビューに分割する

この資料は、scope: product でタグ付けされたルールを表示する専用の「Project rules（プロジェクトルール）」モニタービューを新設し、Guardrails ビューは再利用可能なルール専用に保つことの承認を求めます。

承認のお願い（Requested Approval）

  1. Project rules ビューを新設する。 新しい生成ページ（monitor/project-rules.html＋.ja.html）が、レジストリの scope が product のガードレールを一覧します。現在は6件: Dev Container build、Implementation component contract、Backend lint and tests、Frontend build、Compose config validation、Application monitor freshness。ハードコードではなく scope フィールドで駆動するため、ルールのタグを変えればコード変更なしでビュー間を移動します。
  2. Guardrails を再利用可能ルール専用に保つ。 Guardrails ビューは再利用可能な21件（scope が generic＋bilingual）のみを表示し、プロジェクト固有のものは一覧しなくなります。
  3. モニターのナビに Project rules タブを追加する。 全ページに、英語・日本語の両方で追加します。並びは「広い→具体」の順: Guardrails → Development rules → Project rules — Guardrails は最も広い再利用可能な枠、Development rules は再利用可能な開発特化の層、Project rules はこのプロジェクト固有で汎用化できないルール。

リスク（Risk）

この分割は表示のみです。両ビューは同じレジストリと同じ scope フィールドを読むため、ルールの強制方法は変わりません。レジストリ整合性ガードレール（build-guardrail-monitor.ps1 -Verify）が新ページもカバーするので、レジストリから乖離できません。pwsh scripts/check.ps1 はパスします。

決めないこと（Not Decided Here）

  • 現在の6件の scope: product タグが正しい集合かどうか — 既存ルールがプロジェクト固有か再監査するのは別の後続タスク。
  • 当初の4カテゴリ分類（generic / stack / domain / owner-environment）は導入しない。当面 scope は generic vs product のまま。
  • ルール本文・ゲート・強制は一切書き換えない。
