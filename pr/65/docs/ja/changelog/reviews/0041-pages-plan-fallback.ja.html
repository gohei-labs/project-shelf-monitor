Pages plan fallback

推奨CIPagesToken boundary

判断サマリー

GitHub Pages が利用できない場合でも、monitor を通常の Actions artifact として upload し、Pages deployment だけを skip することで monitor workflow を green に保つ。

オーナーに承認してほしいこと

GitHub Pages が未設定、または現在の account plan で非対応の repository では、artifact fallback を統制された挙動として採用する。

推奨

進めてよい。token は repository admin access を持つ状態になったが、Pages API はこの private repository に対して current plan が Pages 非対応だと返している。workflow は、その外部制約を反映し、赤い状態で止まり続けないようにするべき。

リスクと軽減策

リスク軽減策
monitor の build は成功しているのに main Pages workflow が失敗する。deploy job が Pages availability を確認し、Pages が使えない場合は deployment を skip する。
owner が最新 monitor を確認できなくなる。main run が guardrail-monitor-main-<run_id> を通常の Actions artifact として upload する。
token permission と plan support が混同される。token-management docs で Pages token scopes と repository plan / visibility support を分けて説明する。
deploy skip が本当の build failure を隠す。build job と guardrail checks はこれまで通り失敗する。skip するのは unavailable Pages deployment だけ。

ここで決めないこと

GitHub plan の upgrade、repository visibility の変更、別 hosting provider の選定はここでは決めない。ここでは、それらが workflow の外側の判断として残る間も monitor artifact を確認できる状態にする。

Terminology Check

用語ここでの使い方

Runtime Boundary Check

runtime は GitHub Actions と PowerShell のまま。REST preflight と artifact upload behavior を追加するだけで、新しい runtime、package manager、deployment provider は導入しない。
