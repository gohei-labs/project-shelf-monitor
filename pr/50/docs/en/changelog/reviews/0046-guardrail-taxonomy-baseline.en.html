Guardrail Taxonomy Baseline

This change makes the monitor show what is truly enforced before product development resumes.

Decision Summary

Add a `controlType` taxonomy to the guardrail registry and monitor: hard gate, conditional gate, external control, working agreement, and future control.

What The Owner Needs To Approve

  • Use `controlType` as the visible monitor classification for enforcement strength.
  • Treat backend/frontend product checks as future controls until a component stack is declared.
  • Treat branch protection as a future control for the current private repository plan.
  • Add a stack-neutral component contract gate before implementation directories are introduced.
  • Expose PR preview review-monitor and review-artifact HTML as live URLs the owner can open directly.

Recommendation

Approve this as the baseline before moving into application development. It reduces false confidence and prevents path names from implying implementation stacks.

Risks And Mitigations

RiskMitigation
The monitor becomes more complex.The labels are limited to five stable control types.
Product checks are not executed yet.The new component contract blocks undeclared product component directories; stack-specific execution is added when a stack exists.
Branch protection is still not mechanically enforced.The monitor stops claiming it is active and records it as a future control.
PR comment permissions may hide the preview links.Put live URLs in the workflow summary and PR body. Treat comment failure as non-fatal after preview publish.

Runtime Boundary Check

The authoritative execution lanes remain local `scripts/check.ps1`, CI, and the Dev Container CI lane. This change does not make host-specific Python, Node, pnpm, or Docker availability a product stack requirement.
