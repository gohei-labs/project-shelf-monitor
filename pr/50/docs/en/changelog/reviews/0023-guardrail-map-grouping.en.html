Guardrail Map Grouping

This review asks for approval to group the guardrail map by pillar then category so it stays readable as the registry grows toward 100-200 guardrails, and to make category a registry-enforced field.

Requested Approval

  1. Add a categories map and a category field to every guardrail in playbook/registry.json (10 categories across the 3 pillars).
  2. Regenerate the guardrail map as Pillar then Category collapsible groups with per-group counts, a venue badge per row, and a search box plus venue filter.
  3. scripts/check-registry.ps1 enforces category integrity (present, known, bilingual), so the new field cannot silently drift.

What Changed

AreaRole after this change
playbook/registry.jsonSource of truth; gains a categories map and a per-guardrail category.
scripts/build-guardrail-map.ps1Generates the grouped, collapsible, filterable map from the registry.
scripts/check-registry.ps1Validates category integrity and still blocks any map drift.
playbook/guardrail-map.html / .ja.htmlRegenerated output - do not hand-edit.

Risk

Category names and assignments are an editorial judgment, but they are registry data and can be renamed or re-bucketed without touching the generator. The search and filter are progressive enhancement: with JavaScript disabled the map still renders with all groups open and readable. No gate behavior changes - the same guardrails are enforced, only their presentation is grouped.

Terminology Check

TermMeaning
PillarOne of the three top-level groupings: guardrails, visualization, human-orchestration.
CategoryA finer functional grouping within a pillar (for example review-artifacts, bilingual-docs).
VenueWhere a guardrail runs and whether it blocks: local + CI, CI, conditional, honor, advisory, or external.
Progressive enhancementThe page works without JavaScript; the script only adds search and filtering.

Runtime Boundary Check

This is a documentation, registry, and generator change. It changes how guardrails are displayed, not how or where they run. The venue labels describe existing runtimes (local pre-check via PowerShell, CI, the Dev Container) but introduce none.

The only tooling touched is the PowerShell map generator. Node was used solely to syntax-check (node --check) the embedded filter script during verification; it is not a new runtime dependency. The authoritative environment remains the Dev Container plus Docker Compose.
