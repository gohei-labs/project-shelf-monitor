Guardrail Taxonomy Baseline

product development に戻る前に、monitor が本当に強制されているものを表示できるようにする変更です。

Decision Summary

guardrail registry と monitor に `controlType` taxonomy を追加します。分類は hard gate、conditional gate、external control、working agreement、future control です。

What The Owner Needs To Approve

  • enforcement strength の表示分類として `controlType` を使う。
  • component stack が宣言されるまで、backend/frontend product checks は future controls として扱う。
  • 現在の private repository plan では、branch protection を future control として扱う。
  • implementation directories を追加する前に、stack-neutral な component contract gate を追加する。
  • PR preview では、review monitor と review artifact HTML を owner が直接開ける live URL として表示する。

Recommendation

application development に移る前の baseline として承認することを推奨します。false confidence を減らし、folder name から implementation stack を推測する流れを止められます。

Risks And Mitigations

RiskMitigation
monitor の分類が少し増える。labels は5つの stable control types に限定する。
product checks はまだ実行されない。新しい component contract が、未宣言の product component directory を止める。stack-specific execution は stack が存在してから追加する。
branch protection はまだ機械的に強制されない。monitor では active と表示せず、future control として記録する。
PR comment の権限不足で preview link が表示されない。workflow summary と PR body に live URL を置く。preview publish 後の comment failure は非致命扱いにする。

Runtime Boundary Check

authoritative execution lanes は、local の `scripts/check.ps1`、CI、Dev Container CI lane のままです。この変更は host-specific な Python、Node、pnpm、Docker availability を product stack requirement にしません。
